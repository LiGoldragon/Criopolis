# Devil — rig integration architecture (failure modes)

*Source: bead pc-rv2j, 2026-05-02.*

A. Do not move `~/git/bibliography` into the city in one step. That is the reversible-looking option that is not actually reversible: absolute paths, jj/git remotes, shell scripts, notes, bookmarks, and muscle memory can all keep pointing at the old tree and fail silently or, worse, repopulate it. My preference is option 2 with a migration gate: rename in place to `~/git/library` only after an inventory of references, then register that exact path as a rig. The risk in option 2 is obvious: agents may treat all of `~/git` as writable. Counter it by registering only the exact library path, never the parent, and by putting a sentinel file at the rig root. If visual clarity matters, add a city-side pointer file, not a moved checkout.

B. Start with a single `librarian`, but do not pretend that scales forever. A pack is premature because it creates coordination theater before the library has proven workflows: curator proposes, fetcher downloads, cataloger indexes, and nobody owns the mess when Anna's metadata is wrong. The single-agent failure mode is real, though: once the corpus exceeds the librarian's mental model, stale organization becomes invisible. So make the librarian own an audit report: uncataloged files, duplicate editions, missing source metadata, and failed fetches. Split into a pack only when those audit categories become independent queues with repeated backlog. The devil case against my preference is that the single librarian can become the bottleneck and normalize its own blind spots; audits are the minimum defense.

C. Do not use the abstract rule alone. "City + registered rigs" fails at the exact moment an agent forgets to query the registry. I would use option 3, but generated: materialize a shared boundary block into every prompt from the live rig registry. It should list exact writable paths, exact read-only exclusions, and the command used to verify (`gc rig list`). Add a rule: before any write outside `/home/li/philosophy-city`, the agent must have just verified the target path appears in that materialized allowlist. Unregistered rig writes should not be prevented by trust; they should be made conspicuous. If enforcement exists later, make it path-prefix allowlisting from the same registry. Until then, explicit generated paths beat prose.
