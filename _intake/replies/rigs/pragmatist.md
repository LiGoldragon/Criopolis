# Pragmatist — rig integration architecture

*Source: bead pc-vhwo, 2026-05-02.*

A. Leave the checkout in the `~/git/*` world and register it there after rename: `~/git/library`. Moving a 100-author source library into the city tree buys visual neatness once and charges friction forever: broken shell history, editor workspaces, bookmarks, scripts, citations, and human memory. The rig registry exists exactly to make writable roots explicit without pretending every repo must live under one directory. The prompt boundary should say city plus registered rigs; the path can stay canonical for repos. If Gas City wants a default, keep `city/rigs/<name>` for new rigs. Make library the justified legacy/large-repo exception, not the precedent for random sprawl.

B. Start with one `librarian`. Single is cheaper today and matches the real workflow: add books via `annas`, keep the catalog/index sane, organize files. A pack is warranted only when the work naturally splits by permission, queue, or failure mode. Concrete trigger: curator proposes additions without write/fetch rights; fetcher runs network-heavy `annas` jobs and handles retries; cataloger mutates index/schema and should be reviewable separately. Until those roles produce independent backlogs or need different tool access, a pack is ceremony. Do not add generic `rig-manager` yet; the second or third serious rig will tell us what is generic.

C. Use option 3, but materialize it as explicit text. A fresh agent should not need to grep or remember to run `gc rig list` before knowing where writes are legal. Keep one shared boundary source, generated or maintained from the rig registry, and inject it into every prompt as literal writable roots: `/home/li/philosophy-city/` plus `/home/li/git/library/` when registered. The rule to remember is: "write only in the city root and the listed registered rig roots." Abstract "city + registered rigs" is correct architecture but bad prompt UX; explicit per-template lists drift. Shared source, explicit rendered paths.
