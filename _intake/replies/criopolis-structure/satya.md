## Thread 1 - Sandbox vs sibling, with subsidy

**Position.** The proposition I can vote is: in the short term, Criopolis should be sandboxed from Li's running automation, while a narrow subsidy outside the sandbox performs credentialed host actions on Criopolis's request. I vote for the hybrid-with-subsidy, not for pure sandbox and not for sibling. Pure sibling makes the real-machine deployment surface too broad before role contracts exist. Pure sandbox falsely denies Li's stated constraint that existing automation continue working. The hybrid is true only if the subsidy is a small, logged, capability-shaped bridge, not a second name for full host access.

**Argument.** The explorer brief makes the blast radius factual, not aesthetic: this council is not merely organizing prose; it is approaching active CriomOS and lojix paths that can affect running machines. The mayor rules already classify city lifecycle and relocation as Li-only because they can kill the live system. By analogy, credentialed host action should not be a general property of Criopolis-side agents. The subsidy preserves correspondence to both facts: Criopolis needs a boundary, and Li's automation still needs a way to act.

**Falsification ledger.**

- *decisive* - The subsidy exposes arbitrary shell, broad credential access, or writable host filesystem access to Criopolis-side agents; then the "sandbox" is only nominal.
- *decisive* - A test sandbox cannot read the evidence it needs or cannot produce bead-level work without stale mirrors or manual copying; then the sandbox is false to council work.
- *evidential* - Routine valid tasks require frequent ad hoc subsidy expansions, creating more failure and delay than a sibling model with narrow permissions.
- *evidential* - A staged sibling model proves the same blast-radius limit through explicit OS permissions, dry-run gates, and append-only logs at lower operational cost.

**Vote.** hybrid-with-subsidy / recommendation / strong-for.

## Thread 2 - Roles

**Position.** The first-wave role set should be narrower than the candidate list: code writer, code reviewer, CriomOS deployment specialist, city auditor, and scribe/glossary keeper. Ambassador should defer until the Wasteland research lands. Triage should remain mayor-owned until actual bead volume proves a separate role is needed. Test pilot is a function inside sandboxed implementation and deployment work before it is a separate office.

**Argument.** A role is justified when its object, artifact, write boundary, and sling authority are nameable before the role exists. Otherwise the premise is circular: "create the role because the role would handle what the role handles." The delicate case is the CriomOS deployment specialist. It must be an on-demand role with a dry-run transcript, rollback contract, target declaration, and production veto path; without those, it is merely an implementer with dangerous verbs.

| Role | Provider | Fires | Signature artifact | Workspace boundary | Who can sling |
|---|---|---|---|---|---|
| Code writer / implementer | codex | On-demand after a ratified implementation bead | Patch, changed-file list, test transcript, unresolved-risk note | Declared write set only; default city repos, external repo only when the bead explicitly grants that path | Mayor; Li; council synthesis can authorize |
| Code reviewer | codex | On-demand before merge or after risky patch | Findings with file refs, cited keel rule, severity, and test gap | Read target repo and keel; write review notes only | Mayor; any seat may request through mayor |
| CriomOS deployment specialist | codex | On-demand, never always-on, only for deploy beads | Deploy plan, dry-run transcript, target inventory, rollback contract, subsidy-call transcript | Read CriomOS/lojix axis; write only city-side plans/logs until Li grants a production action | Mayor after council-ratified deploy policy; Li directly in emergency |
| City auditor | codex | Scheduled or on-demand | Health ledger: sessions, beads, mail, dolt size, supervisor status, stuck work | Read city state; write `_intake/audits/` and bead notes | Mayor; scheduled city rule |
| Scribe / glossary keeper | codex | After synthesis or vocabulary correction | Doc patch with source pointers and term changes | `_intake/operating-rules/`, `_intake/glossary.md`, `_intake/replies/`; no `pack.toml` unless separately authorized | Mayor after synthesis; Li |

**Falsification ledger.**

- *decisive* - A proposed role cannot state its artifact or write boundary in advance; then it is not yet a role.
- *decisive* - The deployment specialist can run production-affecting commands without dry-run, rollback, and Li/council gate; then the role violates the reliability constraint.
- *evidential* - Mayor continues to perform most execution work after first-wave roles exist; then the role split is decorative.
- *heuristic* - Two first-wave roles repeatedly produce the same artifact for the same bead; then the taxonomy should be collapsed.

**Vote.** yes on restricted first-wave roles; defer ambassador, triage, and separate test-pilot office / recommendation / strong-for.

## Thread 3 - Coding council and keel

**Position.** The high council should ratify engineering rules; implementing roles should draft, patch, test, review, and report. The code reviewer enforces keel once keel exists, but cannot invent keel. Every new normative keel section, rule, or change of engineering doctrine should be council-ratified. Small wording edits and source-link repairs can be handled by scribe/implementer under mayor synthesis authority if they do not change the rule.

**Argument.** Keel is not just documentation; it is intended to become the engineering rule surface for how criome work is written and reviewed. If reviewer enforcement precedes ratified rules, the reviewer will smuggle private doctrine into "keel compliance." If every comma requires five-seat deliberation, the council becomes an editing bottleneck instead of a truth-bearing ratifier. The boundary is therefore substantive norm versus editorial expression. High council decides the former; implementers handle the latter under visible diffs.

**Falsification ledger.**

- *decisive* - A code reviewer blocks or approves work by citing an unwritten rule as if it were keel; then enforcement has outrun ratification.
- *decisive* - Mayor or implementer changes the practical meaning of a keel rule under "editorial" authority; then the boundary failed.
- *evidential* - Council-ratified keel sections consistently arrive too slowly for real engineering work, without better review quality; then the ratification grain is too fine.
- *heuristic* - Review comments repeatedly expose missing keel rules; then those gaps should become council beads rather than reviewer folklore.

**Vote.** high council ratifies doctrine; implementers execute; reviewer enforces only ratified keel / rule / strong-for.

## Thread 4 - When to start implementing

**Position.** Start now, but only at the lightest viable implementation grain: role-contract templates, one sandboxed test bead for a harmless doc-only implementation, one reviewer pass against that doc patch, and one city-auditor health report. Do not start production deploy capability, subsidy credentials, Wasteland connection, or always-on new roles yet.

**Argument.** Waiting for perfect structure would make "reliability" unfalsifiable; acting on production paths before the guards exist would make "reliability" false. The first implementation move should test the governance loop, not the machines: prompt written, bead-tested, artifact inspected, failure mode recorded, then either ratified or revised. Keel can begin with meta-sections about rule ratification, review authority, and deployment guardrails before it attempts a full engineering doctrine.

**Falsification ledger.**

- *decisive* - The first implementation requires live `pack.toml`, production credentials, city lifecycle commands, or writes outside the declared boundary; then it is premature.
- *decisive* - A role prompt cannot be bead-tested and audited before the role is installed; then role creation should wait.
- *evidential* - Mayor load or stuck work remains high because no execution roles exist; then delay is already harming reliability.
- *heuristic* - The sandbox cannot run a basic bead lifecycle (`bd ready`, reply mirror, note update, close) in isolation; then sandbox implementation work must precede role expansion.

**Vote.** begin with role-contract and harmless sandbox tests; defer production deploy and Wasteland connection / experiment / strong-for.

## Worked example - subsidy-gated deployment

A code writer finishes a CriomOS-related patch and cannot deploy it. The writer files a deploy bead naming target, repo commit, intended command, and rollback assumption. The CriomOS deployment specialist reads the CriomOS and lojix axis, produces a dry-run plan, and asks the subsidy for only a typed operation such as "run this approved dry-run against target X" rather than arbitrary shell. The subsidy returns an append-only transcript. If the dry-run touches undeclared targets, lacks rollback, or asks for credentials beyond the typed operation, the specialist votes no and closes nothing. If it passes, Li or a ratified deploy policy authorizes the production call. The production call returns transcript plus rollback verification. At no point does a general Criopolis agent receive Li's credentials.

What is the subsidy interface: a small set of typed host capabilities with append-only transcripts, or a general command runner with policy around it? My vote depends on the former; the latter refutes the sandbox claim.
