# Notification mechanisms: first-pass survey

Answer first: there is no single primitive that proves "agent A finished" to agent B. The load-bearing pattern appears to be layered: mail is the durable, auditable notification; nudge is the live or deferred attention mechanism; provider hooks are the turn-boundary surface that makes pending mail or queued nudges visible inside an agent session. A completion signal that must survive sleeping, idle, or restarted sessions should not rely on direct nudge alone.

## The three primitives' first-pass shape

Mail is the strongest storage primitive. `gc mail send` creates a message bead and can optionally notify (`--notify`, hidden alias `--nudge`) (`cmd_mail.go:745-785`). Sending records the message first; optional nudge failure is non-fatal and only printed to stderr (`cmd_mail.go:1044-1090`). `gc mail inbox` lists unread messages, `read` displays and marks read, and `peek` displays without marking read (`cmd_mail.go:1148-1251`). `mail check --inject` defaults recipient identity from environment such as `GC_SESSION_ID`, `GC_ALIAS`, or `GC_AGENT` (`cmd_mail.go:131-156`, `249-275`). In hook-inject mode it emits a provider hook context only when unread messages exist, and many failures return exit 0 rather than breaking the hook (`cmd_mail.go:159-219`, `229-247`). Therefore mail is reliable as durable async state, but not as immediate session-visible state unless the receiver polls, receives a `--notify` nudge, or reaches a `UserPromptSubmit` hook.

Nudge is the attention primitive, not the durable source of truth. `gc session nudge <id-or-alias> <message...>` sends text input to a running session and supports `immediate`, `wait-idle`, and `queue`, with `wait-idle` default (`cmd_session.go:1507-1535`, `1619-1630`; `cmd_nudge.go:720-726`). Delivery goes through the runtime provider; `wait-idle` may wait for idle and send a system-reminder, while unsupported/non-idle providers report non-delivery so the command can fall back to queue (`runtime_handle.go:228-269`, `380-413`; `cmd_nudge.go:482-519`). Queued nudges persist under `.gc/runtime/nudges/state.json` with lock-protected state (`state.go:24-50`, `83-146`) and also get backing chore beads labelled `gc:nudge` (`nudge_beads.go:14-17`, `76-119`). Queue items expire after 24 hours (`cmd_nudge.go:1014-1030`) and supersede same agent/source/reference pending or in-flight entries (`cmd_nudge.go:1192-1269`). Silent or soft failures matter: subprocess nudge is an unsupported no-op (`subprocess.go:263-271`), tmux missing session/no server can return nil (`adapter.go:348-386`), hook drain returns 0 on many errors/no-items (`cmd_nudge.go:289-391`), and poller startup discards errors/stdout/stderr (`cmd_nudge.go:814-823`, `932-956`).

Hooks are the surfacing harness. Current Claude-side runtime settings at `/home/li/Criopolis/.gc/settings.json` run `gc prime --hook` on `SessionStart`, `gc nudge drain --inject` plus `gc mail check --inject` on `UserPromptSubmit`, `gc handoff "context cycle"` on `PreCompact`, and a legacy `Stop` hook invoking `gc hook --inject` (`.gc/settings.json:3-52`). Current top-level Codex hooks mirror `SessionStart` and `UserPromptSubmit` and also include `Stop` (`.codex/hooks.json:2-41`); the research rig Codex file has `SessionStart` and `UserPromptSubmit` only (`research/.codex/hooks.json:2-30`). Embedded current defaults for Claude and Codex do not include the legacy Stop hook (`claude.json:1-43`; codex overlay `hooks.json:1-30`). `gc hook --inject` is now silent compatibility behavior and exits 0 without querying work (`cmd_hook.go:18-29`, `53-59`), so Stop should not be treated as a reliable completion notification surface from source alone. Codex hook output is JSON when `--hook-format codex` is used (`hook_output.go:18-51`).
