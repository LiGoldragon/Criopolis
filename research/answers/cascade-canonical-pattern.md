# Cascade canonical pattern research for cr-kbg2ir

## Headline finding

I did not find a current canonical "agent A finishes, Stop hook fires, Stop hook sends `gc mail send --notify B`" pattern. The current Gas City pattern is bead/store driven: create or route durable work, wake a session if needed, and let provider turn hooks surface pending work, mail, and queued nudges. The communication tutorial says the scalable path is: a human mails mayor, mayor slings a bead to the target agent, Gas City routes/wakes a session, agents close beads, and "everyone communicates through the store" rather than direct connections (/git/github.com/gastownhall/gascity/docs/tutorials/04-communication.md:116-126). The `[[agent]]` schema backs that up: default `work_query` finds assigned/routed beads, and default `sling_query` stamps `gc.routed_to=<qualified-name>` so the target's work query can see it (/git/github.com/gastownhall/gascity/internal/config/config.go:1677-1701). `gc mail send --notify` is a mail-plus-nudge primitive, not a workflow dispatcher: the CLI docs say `--notify` nudges after sending (/git/github.com/gastownhall/gascity/cmd/gc/cmd_mail.go:755-762), and the implementation sends the message bead then calls the nudge function if present (/git/github.com/gastownhall/gascity/cmd/gc/cmd_mail.go:1044-1090). The Stop-hook path is explicitly legacy now: the changelog says fresh hook configs no longer install `gc hook --inject` and routed pickup should use SessionStart claim protocol or explicit non-inject `gc hook` (/git/github.com/gastownhall/gascity/CHANGELOG.md:23-26); `gc hook --inject` returns 0 before it resolves or runs a work query (/git/github.com/gastownhall/gascity/cmd/gc/cmd_hook.go:18-39, /git/github.com/gastownhall/gascity/cmd/gc/cmd_hook.go:53-59).

Public docs checked: the public communication tutorial at `https://gascityinc-5c0069dd.mintlify.app/tutorials/04-communication` was checked on 2026-05-04; I used the cloned source copy for line citations.

## $GC_AGENT and env var inventory

Gas City sets these identity/env keys when rendering a session: `GC_SESSION_NAME=<session name>`, `GC_SESSION_ID=<session bead id>`, `GC_TEMPLATE=<template name>`, `GC_SESSION_ORIGIN=ephemeral`, `GC_AGENT=<session name>`, `GC_ALIAS=<qualified agent name>`, `BEADS_ACTOR=<session name>`, `GC_DIR=<work dir>`, `GC_BEADS_SCOPE_ROOT=<city path>`, `GC_RIG`, `GC_RIG_ROOT`, `BEADS_DIR`, and `GT_ROOT=<city path>` (/git/github.com/gastownhall/gascity/cmd/gc/template_resolve.go:223-245). For rig-scoped agents, `GC_RIG`, `GC_RIG_ROOT`, `BEADS_DIR`, and `GC_BEADS_SCOPE_ROOT` are overwritten with the rig values (/git/github.com/gastownhall/gascity/cmd/gc/template_resolve.go:256-260). Gas City also adds city runtime env, `GC_BEADS`, `GC_BIN`, and Dolt env into the same `agentEnv` map (/git/github.com/gastownhall/gascity/cmd/gc/template_resolve.go:246-254).

User/provider env does not win over those generated `GC_*` identities. The final merge order is passthrough env, provider env, agent env, then generated `agentEnv` (/git/github.com/gastownhall/gascity/cmd/gc/template_resolve.go:346-347), and `mergeEnv` says later maps override earlier maps (/git/github.com/gastownhall/gascity/cmd/gc/cmd_start.go:1046-1063).

The runtime receives that env map along with `Command` and `WorkDir` (/git/github.com/gastownhall/gascity/cmd/gc/template_resolve.go:591-603). The tmux runtime then starts from `cfg.Env`, ensures an instance token, and injects runtime hints into the env before session startup (/git/github.com/gastownhall/gascity/internal/runtime/tmux/adapter.go:64-70).

`gc hook` itself resolves target identity from `GC_ALIAS` first, then `GC_AGENT`, and only then template/session fallback or positional argument (/git/github.com/gastownhall/gascity/cmd/gc/cmd_hook.go:61-79). That means a user-authored hook command that inherits the provider process environment can distinguish the firing agent through `GC_ALIAS`/`GC_AGENT`; the Gas City source evidence proves the provider session receives those variables, but provider-specific hook inheritance is outside Gas City's source for each external CLI.

Default hook commands also add some hook-specific env at the command string level. The Claude SessionStart hook exports `GC_MANAGED_SESSION_HOOK=1` and `GC_HOOK_EVENT_NAME=SessionStart` before running `gc prime --hook` (/git/github.com/gastownhall/gascity/internal/hooks/config/claude.json:5-12). The current default UserPromptSubmit hooks run `gc nudge drain --inject` and `gc mail check --inject` without setting `GC_AGENT` themselves, so they rely on the launched session env if they need identity (/git/github.com/gastownhall/gascity/internal/hooks/config/claude.json:27-39).

## Per-agent hook config

Per-agent hook installation exists, but it is not a general per-agent hook-settings path. The `Agent` schema has `work_dir`, `provider`, `start_command`, `args`, `env`, `work_query`, `sling_query`, `install_agent_hooks`, `hooks_installed`, `session_setup`, `session_setup_script`, and `overlay_dir` fields (/git/github.com/gastownhall/gascity/internal/config/config.go:1586-1633, /git/github.com/gastownhall/gascity/internal/config/config.go:1677-1750). I did not find `settings_path`, `hooks_dir`, `extra_args`, or `command_override` fields in the agent/provider schemas; the supported equivalents are `start_command`, `args`, `env`, `overlay_dir`, and provider `command`/`args`/`env`/`path_check` (/git/github.com/gastownhall/gascity/internal/config/config.go:1614-1633, /git/github.com/gastownhall/gascity/internal/config/provider.go:56-80).

`install_agent_hooks` can be set at workspace or agent level. The tutorial shows workspace `install_agent_hooks = ["claude"]` and agent-local `install_agent_hooks = ["claude"]` (/git/github.com/gastownhall/gascity/docs/tutorials/04-communication.md:136-151). Source says the agent-level value replaces the workspace value, not adds to it (/git/github.com/gastownhall/gascity/internal/config/config.go:1709-1711, /git/github.com/gastownhall/gascity/internal/config/resolve.go:103-114).

`hooks_installed = true` is the source-supported way to tell Gas City hooks already exist and suppress automatic "no hooks" startup behavior (/git/github.com/gastownhall/gascity/internal/config/config.go:1721-1727). `AgentHasHooks` treats explicit `hooks_installed` as highest priority, then treats Claude-family providers as hook-enabled, then checks whether the provider name is in resolved `install_agent_hooks` (/git/github.com/gastownhall/gascity/internal/config/resolve.go:573-606).

Hook installation only supports known provider families. Claude writes city-level settings; Codex/Gemini/OpenCode/Copilot/Cursor/Pi/OMP install overlay-managed files under the agent work dir (/git/github.com/gastownhall/gascity/internal/hooks/hooks.go:101-125). Tmux startup copies pack and agent overlays into `cfg.WorkDir` before creating the session, using both the provider name and `install_agent_hooks` as overlay provider slots (/git/github.com/gastownhall/gascity/internal/runtime/tmux/adapter.go:79-95). That can support per-agent files only if the chosen `WorkDir` and overlay strategy do not cause agents to overwrite each other's provider settings.

## Wrapper-executable approach

A wrapper provider is feasible in the source model. Providers are user-definable and support `command`, `args`, `env`, and `path_check`; `path_check` exists specifically for wrapper commands where the PATH check should validate the real binary instead of the wrapper command (/git/github.com/gastownhall/gascity/internal/config/provider.go:37-80). Agents can select a provider with `provider`, override the command with `start_command`, override args with `args`, and add env with `env` (/git/github.com/gastownhall/gascity/internal/config/config.go:1614-1633).

Launch composition starts from the resolved provider command, applies option defaults/overrides, and appends a provider-owned settings file if present (/git/github.com/gastownhall/gascity/internal/config/launch_command.go:21-55). The resolved command string is literally provider command plus shell-quoted args (/git/github.com/gastownhall/gascity/internal/config/provider.go:210-215). The runtime config then carries `Command`, `Env`, and `WorkDir` together into the runtime (/git/github.com/gastownhall/gascity/cmd/gc/template_resolve.go:591-603). So a wrapper can preserve the city working directory by leaving `work_dir` alone or setting it explicitly, while changing only `provider`/`start_command`/`args`/`env`.

The sharp edge: automatic `--settings` append is currently literal-Claude only in the launch path. `appendProviderSettings` appends `--settings <path>` when `ProviderSettingsSource` returns a file (/git/github.com/gastownhall/gascity/internal/config/launch_command.go:81-91), and `ProviderSettingsSource` returns nothing unless `providerName == "claude"` (/git/github.com/gastownhall/gascity/internal/config/launch_command.go:94-100). This differs from hook detection, which recognizes custom providers whose built-in family is Claude (/git/github.com/gastownhall/gascity/internal/config/resolve.go:581-596). A wrapper named something other than `claude` may therefore need to pass provider-specific settings/config flags itself.

## Examples found in the wild

The official tutorial pattern is mail/nudge for wake-up and `gc sling` for work routing. It says a nudge only wakes a turn; hooks deliver pending mail at the next turn (/git/github.com/gastownhall/gascity/docs/tutorials/04-communication.md:80-96). It then shows the mayor routing work by running `gc sling my-project/reviewer`, and states that Gas City wakes and routes sessions while work persists in the store (/git/github.com/gastownhall/gascity/docs/tutorials/04-communication.md:98-126).

The checked-in Gastown example pack says `gc nudge` is the core mechanism for inter-agent communication, gives `gc mail send ... --notify` as a common pattern, and warns that nudging does not assign work; actual work arrives through bead assignment or pool routing (/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/assets/prompts/crew.template.md:196-220). The same prompt says raw `tmux send-keys` is unreliable and queued nudges arrive through `UserPromptSubmit` hooks (/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/assets/prompts/crew.template.md:235-257).

The Gastown example pack's "operational awareness" fragment says routine protocol signals should be nudges, not mail, because bead state is the durable record (/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/template-fragments/operational-awareness.template.md:90-100). Its propulsion fragment says startup behavior is: check assigned work, execute if hooked, else run `WorkQuery`, else check mail (/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/template-fragments/propulsion.template.md:32-40, /git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/template-fragments/propulsion.template.md:79-87).

The `gascity-packs` MCP agent mail pack is another prompt-bound inbox pattern, not a Stop cascade. Its README says it provides an MCP server, `/coordinate` skill, and a `UserPromptSubmit` hook that checks the inbox on each prompt (/git/github.com/gastownhall/gascity-packs/flywheel/mcp-agent-mail/README.md:1-10). Its Claude overlay contains only a `UserPromptSubmit` command of `am inbox --count`, not a Stop hook (/git/github.com/gastownhall/gascity-packs/flywheel/mcp-agent-mail/overlay/.claude/settings.json:10-17).

Orders are the source-supported alternative for controller-driven cascades. The orders tutorial says an order pairs a trigger with a formula or shell script and the controller checks triggers automatically every tick (/git/github.com/gastownhall/gascity/docs/tutorials/07-orders.md:7-15). Event orders can fire on `bead.closed` and use cursor tracking so the same event is not processed twice (/git/github.com/gastownhall/gascity/docs/tutorials/07-orders.md:198-213). The dispatcher source says due orders dispatch as wisps or exec scripts and create a tracking bead before launching the action to prevent re-fire on the next tick (/git/github.com/gastownhall/gascity/cmd/gc/order_dispatch.go:26-32). Manual orders are explicitly filtered from auto-dispatch (/git/github.com/gastownhall/gascity/cmd/gc/order_dispatch.go:102-110).

Beads itself is not the mail/orchestration plane in current source. The `bd mail` command says the mail implementation is provided by the orchestrator, not by Beads itself (/git/github.com/gastownhall/beads/cmd/bd/mail.go:14-21). That aligns with Gas City's use of Beads as durable work state and Gas City as the coordination layer.

## Stop hook patterns

Current fresh managed hooks do not use Stop for cascades. The default Claude settings include `SessionStart`, `PreCompact`, and `UserPromptSubmit` only (/git/github.com/gastownhall/gascity/internal/hooks/config/claude.json:4-43). The default Codex overlay includes `SessionStart` and `UserPromptSubmit` only (/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/overlay/per-provider/codex/.codex/hooks.json:1-30). Tests assert fresh Claude settings should contain `gc nudge drain --inject` and should not contain `gc hook --inject` (/git/github.com/gastownhall/gascity/internal/hooks/hooks_test.go:124-132), and fresh overlay-managed provider files should not install no-op `gc hook --inject` (/git/github.com/gastownhall/gascity/internal/hooks/hooks_test.go:776-783).

The implementation makes `gc hook --inject` non-dispatching. The CLI help says `--inject` is "silent legacy Stop-hook compatibility" and skips the work query (/git/github.com/gastownhall/gascity/cmd/gc/cmd_hook.go:18-39). The command returns 0 immediately in inject mode (/git/github.com/gastownhall/gascity/cmd/gc/cmd_hook.go:53-55), and tests assert inject mode does not run the work query, is silent, and does not create a work-query marker (/git/github.com/gastownhall/gascity/cmd/gc/cmd_hook_test.go:100-181). The beads tutorial says the same thing: legacy Stop-hook `gc hook --inject` is silent compatibility behavior and no longer injects work into the agent (/git/github.com/gastownhall/gascity/docs/tutorials/06-beads.md:382-388).

There is historical evidence that Stop hooks used to be considered for continued work pickup. An archived feature-parity note says Codex hook installation used Stop with `gc hook --inject` for queued work (/git/github.com/gastownhall/gascity/engdocs/archive/analysis/feature-parity.md:348-354), and an archived provider parity audit says missing Stop meant `gc hook --inject` never ran and agents sat idle after completing work (/git/github.com/gastownhall/gascity/engdocs/archive/analysis/non-claude-provider-parity-audit.md:110-117). Current changelog/source/tests supersede that archived assumption by making inject mode a no-op and removing it from fresh configs (/git/github.com/gastownhall/gascity/CHANGELOG.md:23-26, /git/github.com/gastownhall/gascity/internal/hooks/hooks_test.go:130-132).

One doc/code mismatch: the communication tutorial says managed Claude settings fire commands "on shutdown" (/git/github.com/gastownhall/gascity/docs/tutorials/04-communication.md:153-157), but the current checked-in Claude settings file has no Stop/SessionEnd hook (/git/github.com/gastownhall/gascity/internal/hooks/config/claude.json:4-43), and the current changelog says fresh configs no longer install the Stop/session-end `gc hook --inject` path (/git/github.com/gastownhall/gascity/CHANGELOG.md:23-26).

## Open questions / things we still do not understand

I did not find a source-backed community convention for "always mail+notify the next agent on close." The source-backed convention is "make the next work visible in beads; use nudge/mail to wake or inform; rely on turn hooks to surface pending state" (/git/github.com/gastownhall/gascity/docs/tutorials/04-communication.md:116-126, /git/github.com/gastownhall/gascity/internal/config/config.go:1677-1701, /git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/template-fragments/operational-awareness.template.md:90-100).

I did not find a built-in "next agent on bead close" notification policy. The migration guide says `gt notify` has no direct equivalent and notification policy is not a top-level SDK command family (/git/github.com/gastownhall/gascity/docs/getting-started/coming-from-gastown.md:455-463). The closest source-supported automatic mechanism is an order trigger such as `on = "bead.closed"` (/git/github.com/gastownhall/gascity/docs/tutorials/07-orders.md:198-213).

The wrapper approach has a source-level mismatch to verify before relying on it: hook detection recognizes custom Claude-family providers, but automatic `--settings` append currently checks only the literal provider name `claude` (/git/github.com/gastownhall/gascity/internal/config/resolve.go:581-596, /git/github.com/gastownhall/gascity/internal/config/launch_command.go:94-100). A wrapper can still pass its own settings/config flags, but that is wrapper-owned behavior.

I did not find source evidence that Gas City itself controls external provider hook-command environment inheritance beyond setting the launched session environment. The relevant Gas City evidence is that `GC_AGENT`/`GC_ALIAS` are in runtime env (/git/github.com/gastownhall/gascity/cmd/gc/template_resolve.go:223-260) and `gc hook` reads those vars (/git/github.com/gastownhall/gascity/cmd/gc/cmd_hook.go:61-79); whether a specific provider's Stop hook inherits them is controlled by that provider's hook runner.
