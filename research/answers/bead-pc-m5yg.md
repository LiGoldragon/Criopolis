## Part A - The Wasteland

### Finding

I found the source at `gastownhall/wasteland` (`https://github.com/gastownhall/wasteland`). The CLI source names `wl` as "the Wasteland CLI - federation protocol for Gas Towns" and the README says Wasteland is "Federation protocol for Gas Towns" with a reference commons at `hop/wl-commons` (`https://www.dolthub.com/repositories/hop/wl-commons`). `/git/github.com/gastownhall/wasteland/cmd/wl/main.go:1-2`, `/git/github.com/gastownhall/wasteland/README.md:1-9`

The implementation-level definition is in `internal/federation`: Wasteland is "a federation of Gas Towns via DoltHub"; each rig has a fork of a shared commons database; rigs register in the `rigs` table; wanted work and completions move through DoltHub fork/PR/merge primitives. `/git/github.com/gastownhall/wasteland/internal/federation/federation.go:1-7`

I did not find a production `gc wasteland` subcommand in Gas City. The `gc` root command list contains commands such as `start`, `rig`, `mail`, `order`, `pack`, `sling`, `beads`, `workflow`, `bd`, and `dolt-state`, but no Wasteland command. `/git/github.com/gastownhall/gascity/cmd/gc/main.go:153-204` The archived image-dependency design explicitly lists `wl` as "Wasteland CLI (not yet included)" and sketches copying `/out/wl` into an agent image, which is integration planning, not current `gc` command integration. `/git/github.com/gastownhall/gascity/engdocs/archive/designs/image-dependency-versioning.md:30-32`, `/git/github.com/gastownhall/gascity/engdocs/archive/designs/image-dependency-versioning.md:55-70`

### Protocol / data model

The primary protocol is Dolt/DoltHub around a shared SQL schema. The schema contains `_meta`, `rigs`, `wanted`, `completions`, `stamps`, `badges`, `boot_blocks`, `chain_meta`, and `rig_links`. `/git/github.com/gastownhall/wasteland/schema/commons.sql:1-115` The core `commons.DB` interface abstracts SQL query/exec, branch listing, branch deletion, push, sync, merge, remote branch deletion, and `PushWithSync`; the comment says it is implemented by `backend.LocalDB` and `backend.RemoteDB`. `/git/github.com/gastownhall/wasteland/internal/commons/commons.go:20-57`

The remote backend speaks the DoltHub REST API at `https://www.dolthub.com/api/v1alpha1`: reads from `main` go to the upstream shared database, branch reads and writes go to the user's fork, SQL reads use a GET query endpoint, and DML writes use DoltHub's write API. `/git/github.com/gastownhall/wasteland/internal/backend/remote.go:20-25`, `/git/github.com/gastownhall/wasteland/internal/backend/remote.go:71-121` Remote write API pushes are no-ops locally because the API auto-pushes; remote wild-west mode is rejected because the REST API cannot push fork-to-upstream. `/git/github.com/gastownhall/wasteland/internal/backend/remote.go:198-216`

The provider abstraction has DoltHub as production plus `file://`, bare git, and GitHub-style providers for alternate/test remotes. The interface exposes `DatabaseURL`, `Fork`, `CreatePR`, and `Type`. `/git/github.com/gastownhall/wasteland/internal/remote/remote.go:1-20` The Wasteland config also stores a `HopURI`, described as an HOP protocol URI, but in the source I read it is stored as identity/link metadata rather than used as a socket-level transport. `/git/github.com/gastownhall/wasteland/internal/federation/federation.go:57-61`, `/git/github.com/gastownhall/wasteland/internal/federation/federation.go:312-324`

The visible user interfaces are the `wl` CLI and an HTTP API/web UI. The CLI root adds commands for create/join/post/claim/done/pending/accept/reject/close/update/delete/browse/status/sync/review/approve/merge/TUI/serve/leaderboard/profile and related operations. `/git/github.com/gastownhall/wasteland/cmd/wl/main.go:57-127` The API server wraps `sdk.Client` and exposes browse, detail, dashboard, mutation, and branch operations as JSON. `/git/github.com/gastownhall/wasteland/internal/api/server.go:1-5` Its routes include wanted-board reads, wanted mutations, branch apply/discard/PR/diff, public scoreboard/profile, settings, and sync. `/git/github.com/gastownhall/wasteland/internal/api/routes.go:3-52`

### Joining

`wl join` defaults to upstream `hop/wl-commons`. Its help text says the default mode is remote DoltHub API mode, with `--local-db` for a local Dolt clone, and its documented steps are: fork the upstream commons, register the rig in `rigs`, open a PR for registration, and save local config. `/git/github.com/gastownhall/wasteland/cmd/wl/cmd_join.go:19-32`, `/git/github.com/gastownhall/wasteland/cmd/wl/cmd_join.go:69-97`

Remote join requires `DOLTHUB_TOKEN` and `DOLTHUB_ORG`, uses the fork org as the default handle, forks upstream via the provider, writes registration SQL to branch `wl/register/<handle>`, creates a PR, and saves config with `BackendRemote`, `ModePR`, rig handle, fork, upstream URL, and HOP URI. `/git/github.com/gastownhall/wasteland/cmd/wl/cmd_join.go:242-350`

Local join orchestrates the same protocol with a local Dolt clone: parse upstream, fork unless direct, clone the fork, add upstream remote, branch `wl/register/<handle>`, register the rig, push the branch, open a PR, and save config in PR mode. `/git/github.com/gastownhall/wasteland/internal/federation/federation.go:224-330` The registration SQL upserts the rig into `rigs`, sets `trust_level` to `1`, and records the `hop_uri`. `/git/github.com/gastownhall/wasteland/internal/commons/registration.go:5-25`

### What a city sees

A city or rig sees a shared wanted board plus identity and reputation records: `wanted` rows contain title, description, project, type, priority, tags, poster, claimer, status, effort, evidence, and sandbox fields; `completions` records completion evidence and validation; `stamps` record reputation/validation with valence, confidence, severity, skill tags, message, and a database check preventing `author = subject`. `/git/github.com/gastownhall/wasteland/schema/commons.sql:22-72`

The lifecycle is work-board shaped: `InsertWantedDML` creates open wanted items, `ClaimWantedDML` changes `open` to `claimed`, `SubmitCompletionDML` moves a claimed item to `in_review` and inserts a completion, and `AcceptCompletionDML` inserts a stamp, validates the completion, and marks the item `completed`. `/git/github.com/gastownhall/wasteland/internal/commons/commons.go:275-332`, `/git/github.com/gastownhall/wasteland/internal/commons/commons.go:364-380`, `/git/github.com/gastownhall/wasteland/internal/commons/commons.go:679-706`

The SDK is mode-aware. In wild-west mode, mutation DML runs on main and pushes; in PR mode, mutation DML runs on `wl/<rig>/<wanted-id>` branches, pushes that branch, cleans up reverted branches, and can auto-create a PR. `/git/github.com/gastownhall/wasteland/internal/sdk/mutate.go:19-108` The high-level SDK exposes operations for claim, unclaim, done, accept, reject, close, delete, post, and update. `/git/github.com/gastownhall/wasteland/internal/sdk/mutations.go:38-124`, `/git/github.com/gastownhall/wasteland/internal/sdk/mutations.go:321-375`

### Governance

PR mode is the default. `ResolveMode` defaults empty mode to `pr`, and the README describes PR mode as pushing to a fork and opening PRs upstream, while wild-west pushes directly to upstream/origin for maintainers with write access. `/git/github.com/gastownhall/wasteland/internal/federation/federation.go:84-90`, `/git/github.com/gastownhall/wasteland/README.md:239-304`

The lifecycle rules are encoded in `commons`: claim is `open -> claimed`, done is `claimed -> in_review`, accept/close are `in_review -> completed`, reject is `in_review -> claimed`, and delete is `open -> withdrawn`. `/git/github.com/gastownhall/wasteland/internal/commons/lifecycle.go:11-39` Permission rules allow any rig to claim, only claimer/poster to unclaim, only claimer to mark done, poster/admin to accept/reject/close, and only poster to delete; hardcoded admins are `julianknutsen`, `steveyegge`, and `csells`. `/git/github.com/gastownhall/wasteland/internal/commons/lifecycle.go:137-170` Accept also blocks self-stamping and tells the worker to use `wl close` instead. `/git/github.com/gastownhall/wasteland/internal/sdk/mutations.go:91-123`

## Part B - Community gas-city structures

### Search surface

I checked the Gas City README and examples, `gastownhall` org repos, existing `/git/github.com/...` clones, GitHub code search, and web search. GitHub code search for public non-`gastownhall` `city.toml`-shaped configs found two useful hits: `bmt/gascity-explore` (`https://github.com/bmt/gascity-explore`) and `ocasazza/jump-cannon` (`https://github.com/ocasazza/jump-cannon`). A broader code search for `"pack.toml" "gascity"` returned no additional public pack-shaped hits in that pass.

Gas City itself frames the surface: it is an orchestration-builder SDK with declarative `city.toml`, multiple runtime providers, beads-backed work tracking, a controller loop, and packs/overrides/rig-scoped orchestration. `/git/github.com/gastownhall/gascity/README.md:3-22` Its repo map points at `examples/` for sample cities/packs/configs. `/git/github.com/gastownhall/gascity/README.md:99-109`

### Official examples in `gastownhall/gascity`

`examples/gastown` is Gas Town expressed as Gas City config. It imports a `gastown` pack, uses workspace provider `claude`, declares city-scoped mayor/deacon/boot/dog and rig-scoped witness/refinery/polecat, and requires rigs to be registered with `gc rig add <path>`. `/git/github.com/gastownhall/gascity/examples/gastown/city.toml:1-25`, `/git/github.com/gastownhall/gascity/examples/gastown/city.toml:39-60` Its named sessions are mayor/deacon/boot as city `always`, witness as rig `always`, and refinery as rig `on_demand`. `/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/pack.toml:32-55` Agent definitions show city-scoped mayor/deacon/boot, rig-scoped witness, rig-scoped refinery in an isolated worktree, and a rig-scoped polecat pool with `max_active_sessions = 5`. `/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/agents/mayor/agent.toml:1-6`, `/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/agents/deacon/agent.toml:1-6`, `/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/agents/boot/agent.toml:1-4`, `/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/agents/witness/agent.toml:1-6`, `/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/agents/refinery/agent.toml:1-7`, `/git/github.com/gastownhall/gascity/examples/gastown/packs/gastown/agents/polecat/agent.toml:1-8`

`examples/swarm` is the contrasting flat design. It uses provider `claude`, city agents mayor/deacon/dog, rig agents coder/committer, shared rig directories, no worktree isolation, no formulas, and flat peer coordination rather than hierarchical dispatch. `/git/github.com/gastownhall/gascity/examples/swarm/city.toml:1-28` Its pack defines named sessions for mayor city `on_demand`, deacon city `always`, and committer rig `on_demand`; coders are a rig pool with `min_active_sessions = 1` and `max_active_sessions = 5`. `/git/github.com/gastownhall/gascity/examples/swarm/packs/swarm/pack.toml:1-29`, `/git/github.com/gastownhall/gascity/examples/swarm/packs/swarm/agents/coder/agent.toml:1-5`

`examples/lifecycle` is a deterministic demo with no city agents: all agents are rig-scoped, polecats work in isolated worktrees/feature branches, and refinery merges to main. `/git/github.com/gastownhall/gascity/examples/lifecycle/city.toml:1-22` Its pack declares only a refinery rig `on_demand` named session; both polecat and refinery use `start_command` bash scripts rather than a model provider. `/git/github.com/gastownhall/gascity/examples/lifecycle/packs/lifecycle/pack.toml:1-16`, `/git/github.com/gastownhall/gascity/examples/lifecycle/packs/lifecycle/agents/polecat/agent.toml:1-6`, `/git/github.com/gastownhall/gascity/examples/lifecycle/packs/lifecycle/agents/refinery/agent.toml:1-4`

`examples/dolt` and `examples/bd` are infrastructure packs rather than cities. `dolt` provides lifecycle commands for the Dolt SQL server backing bead storage, and `bd` imports `dolt`, ships bd/beads integration, and is active by default for the SDK's default beads provider. `/git/github.com/gastownhall/gascity/examples/dolt/pack.toml:1-10`, `/git/github.com/gastownhall/gascity/examples/bd/pack.toml:1-17`

## What this suggests for Criopolis

Wasteland's relevant pattern is not "cities chatting" in a direct-message sense; in the source, the durable inter-city contract is a shared Dolt commons with registration, wanted work, completions, stamps, branches, and PRs. That matters for a council structure because it separates proposal/visibility from acceptance: work can be posted and claimed broadly, but default PR mode means upstream state changes can require review before becoming canonical. `/git/github.com/gastownhall/wasteland/internal/federation/federation.go:1-7`, `/git/github.com/gastownhall/wasteland/internal/sdk/mutate.go:19-108`, `/git/github.com/gastownhall/wasteland/internal/commons/lifecycle.go:145-170`

The community examples show two pressures that map onto Criopolis. Official `gastown` centralizes reliability through mayor/deacon/boot/witness/refinery/polecat, while `swarm` deliberately reduces hierarchy and lets rig coders self-organize through beads/mail. `/git/github.com/gastownhall/gascity/examples/gastown/city.toml:1-20`, `/git/github.com/gastownhall/gascity/examples/swarm/city.toml:1-12` `bmt/gascity-explore` is the clearest third-party adaptation: it removes direct refinery merge, inserts steward/reviewer/human PR gates, and documents practical pain around formulas, custom vars, provider pools, rig-scoped pools, and startup timeouts. `/git/github.com/bmt/gascity-explore/README.md:28-44`, `/git/github.com/bmt/gascity-explore/README.md:193-218`

For Criopolis, the visible patterns suggest keeping reliability roles separate from authority roles. Wasteland-style PR/default-review semantics and Kit-style steward/reviewer gates are relevant to the council's veto concern; Jump Cannon's pinned-vs-auto provider split is relevant to Li's reliability/adaptability concern; and the mayor-doing-too-much concern maps to moving routine health, review, and merge/state-transition work into named specialist agents rather than letting the mayor be the only cross-cutting control point. `/git/github.com/ocasazza/jump-cannon/city.toml:13-35`, `/git/github.com/ocasazza/jump-cannon/agents/mayor/agent.toml:1-5`, `/git/github.com/ocasazza/jump-cannon/agents/deacon/agent.toml:1-5`, `/git/github.com/bmt/gascity-explore/packs/kit/agents/steward/agent.toml:1-8`, `/git/github.com/bmt/gascity-explore/packs/kit/agents/reviewer/agent.toml:1-8`
