# Implementation Survey

## AutoGPT

AutoGPT classic is primarily an autonomous action loop. `propose_action` resets trace, gathers directive resources/constraints/best practices, gathers commands, prepares messages, builds a prompt with task/profile/directives/commands, calls `complete_and_parse`, and increments cycle count (`/git/github.com/Significant-Gravitas/AutoGPT/classic/original_autogpt/autogpt/agents/agent.py:266-339`). `complete_and_parse` calls the configured LLM provider with messages, model name, parser, and function specs, then runs an after-parse pipeline (`/git/github.com/Significant-Gravitas/AutoGPT/classic/original_autogpt/autogpt/agents/agent.py:341-371`). `execute` checks permissions, executes one tool directly or multiple tools in parallel, handles errors, records ReWOO execution results, and runs after-execute (`/git/github.com/Significant-Gravitas/AutoGPT/classic/original_autogpt/autogpt/agents/agent.py:373-460`). `_execute_tools_parallel` uses `asyncio.gather` and returns combined success, partial success, or error results (`/git/github.com/Significant-Gravitas/AutoGPT/classic/original_autogpt/autogpt/agents/agent.py:499-557`). Strategy selection includes `rewoo`, `plan_execute`, `reflexion`, `tree_of_thoughts`, and `lats` (`/git/github.com/Significant-Gravitas/AutoGPT/classic/original_autogpt/autogpt/agents/agent.py:588-633`).

Architectural take: AutoGPT is a strong reference for tool-loop mechanics and strategy swapping, not for a deliberative council.

## BabyAGI

The historical BabyAGI script is a compact task-queue agent. `task_creation_agent` creates new tasks from the prior result and incomplete tasks (`/git/github.com/yoheinakajima/babyagi`, commit `a37ca09`, `classic/babyagi.py:46-50`). `prioritization_agent` rewrites and reprioritizes the queue (`/git/github.com/yoheinakajima/babyagi`, commit `a37ca09`, `classic/babyagi.py:52-69`). `execution_agent` retrieves vector context and asks the LLM to perform one task (`/git/github.com/yoheinakajima/babyagi`, commit `a37ca09`, `classic/babyagi.py:70-84`). `context_agent` queries Pinecone and returns sorted prior task metadata (`/git/github.com/yoheinakajima/babyagi`, commit `a37ca09`, `classic/babyagi.py:86-94`). The main loop pops a task, executes it, stores the result in Pinecone, creates new tasks, appends them, and reprioritizes (`/git/github.com/yoheinakajima/babyagi`, commit `a37ca09`, `classic/babyagi.py:96-136`).

The current BabyAGI repo has shifted toward function registration and tool loops. `register_function` captures source code, metadata, imports, dependencies, key dependencies, and triggers before registering the function, and the wrapper executes through a function executor (`/git/github.com/yoheinakajima/babyagi/babyagi/functionz/core/registration.py:16-43`). Its `react_agent` draft builds LiteLLM tool definitions from registered functions, loops up to five iterations, executes requested functions, avoids duplicate calls, and returns reasoning path, function history, and final answer (`/git/github.com/yoheinakajima/babyagi/babyagi/functionz/packs/drafts/react_agent.py:28-190`).

Architectural take: BabyAGI's durable lesson is queue discipline: explicit backlog, task creation, prioritization, execution, memory.

## MetaGPT

MetaGPT encodes the "software company" metaphor in source. `Team` is documented as one or more roles, SOPs, and an environment for multi-agent activity (`/git/github.com/geekan/MetaGPT/metagpt/team.py:32-36`). `Team.__init__` creates an `Environment` or `MGXEnv`, hires supplied roles, and can set an environment description (`/git/github.com/geekan/MetaGPT/metagpt/team.py:45-57`). `hire` adds roles to the environment (`/git/github.com/geekan/MetaGPT/metagpt/team.py:83-86`). `run_project` publishes the human requirement as a message (`/git/github.com/geekan/MetaGPT/metagpt/team.py:102-108`). `run` loops until rounds are exhausted or the environment is idle, checks budget, and runs the environment (`/git/github.com/geekan/MetaGPT/metagpt/team.py:123-138`).

The environment hosts roles and history (`/git/github.com/geekan/MetaGPT/metagpt/environment/base_env.py:124-135`). It adds roles and injects environment/context (`/git/github.com/geekan/MetaGPT/metagpt/environment/base_env.py:156-173`). It publishes messages to recipients and stores history (`/git/github.com/geekan/MetaGPT/metagpt/environment/base_env.py:175-195`). It runs all non-idle roles concurrently with `asyncio.gather` (`/git/github.com/geekan/MetaGPT/metagpt/environment/base_env.py:197-212`).

Architectural take: MetaGPT is a strong reference for role/SOP and message environment design. Criopolis differs because its role set is evaluative and civic, not production-line software-company roles.

## CrewAI

CrewAI exposes `sequential` and `hierarchical` process types, with `consensual` present only as a commented placeholder (`/git/github.com/crewAIInc/crewAI/lib/crewai/src/crewai/process.py:4-11`). A hierarchical process requires `manager_llm` or `manager_agent`, and the manager agent may not be included in the crew's ordinary agent list (`/git/github.com/crewAIInc/crewAI/lib/crewai/src/crewai/crew.py:627-649`). Sequential process executes tasks directly, while hierarchical process creates a manager then executes tasks (`/git/github.com/crewAIInc/crewAI/lib/crewai/src/crewai/crew.py:1397-1404`). If no manager agent is supplied, CrewAI creates one with a hierarchical manager role/goal/backstory, delegation tools over the agents, `allow_delegation=True`, and the manager LLM (`/git/github.com/crewAIInc/crewAI/lib/crewai/src/crewai/crew.py:1406-1431`). `_execute_tasks` processes tasks, starts async tasks when marked, joins pending async futures before sync tasks, and returns crew output (`/git/github.com/crewAIInc/crewAI/lib/crewai/src/crewai/crew.py:1441-1508`). Manager delegation resolves an agent by role, creates a task for the selected agent, and calls `selected_agent.execute_task` (`/git/github.com/crewAIInc/crewAI/lib/crewai/src/crewai/tools/agent_tools/base_agent_tools.py:70-125`).

Architectural take: CrewAI is the most direct manager-worker product pattern in the survey.

## Microsoft AutoGen

AutoGen AgentChat is runtime-centered group coordination. The base group chat manager handles start by checking termination, validating state, publishing start messages to output/group topics, appending messages to the thread, checking termination, and selecting next speakers (`/git/github.com/microsoft/autogen/python/packages/autogen-agentchat/src/autogen_agentchat/teams/_group_chat/_base_group_chat_manager.py:86-132`). On agent response, it appends inner and chat messages to the thread, removes the active speaker, waits until active speakers are empty, applies termination, and transitions to the next speakers (`/git/github.com/microsoft/autogen/python/packages/autogen-agentchat/src/autogen_agentchat/teams/_group_chat/_base_group_chat_manager.py:134-164`). Speaker transition calls `select_speaker`, validates selected names, logs selection, publishes request messages to selected speaker topics, and appends active speakers (`/git/github.com/microsoft/autogen/python/packages/autogen-agentchat/src/autogen_agentchat/teams/_group_chat/_base_group_chat_manager.py:172-193`).

AutoGen also ships policy variants. Round-robin group chat stores a next-speaker index and selects modulo participant count (`/git/github.com/microsoft/autogen/python/packages/autogen-agentchat/src/autogen_agentchat/teams/_group_chat/_round_robin_group_chat.py:46-82`). Selector group chat can use a selector function, candidate function, model-based selection, and repeated-speaker avoidance (`/git/github.com/microsoft/autogen/python/packages/autogen-agentchat/src/autogen_agentchat/teams/_group_chat/_selector_group_chat.py:152-217`). Swarm group chat selects the next speaker based on the last handoff message or current speaker (`/git/github.com/microsoft/autogen/python/packages/autogen-agentchat/src/autogen_agentchat/teams/_group_chat/_swarm_group_chat.py:45-98`). Magentic-One orchestrator maintains facts, plan, and progress ledgers and uses the ledger to decide completion, stalls, loops, replanning, and next speaker (`/git/github.com/microsoft/autogen/python/packages/autogen-agentchat/src/autogen_agentchat/teams/_group_chat/_magentic_one/_magentic_one_orchestrator.py:134-190`, `/git/github.com/microsoft/autogen/python/packages/autogen-agentchat/src/autogen_agentchat/teams/_group_chat/_magentic_one/_magentic_one_orchestrator.py:300-450`). Society-of-Mind wraps an inner team and synthesizes its output into a final response (`/git/github.com/microsoft/autogen/python/packages/autogen-agentchat/src/autogen_agentchat/agents/_society_of_mind_agent.py:168-250`).

Architectural take: AutoGen is the deepest source reference for group chat mechanics, speaker policy, and nested teams.

## LangGraph

LangGraph is a low-level graph orchestration substrate. `StateGraph` nodes communicate by reading and writing shared state, and reducers aggregate values received from multiple nodes (`/git/github.com/langchain-ai/langgraph/libs/langgraph/langgraph/graph/state.py:119-133`). `add_edge` defines directed edges and supports waiting for all start nodes before executing an end node (`/git/github.com/langchain-ai/langgraph/libs/langgraph/langgraph/graph/state.py:837-889`). `add_conditional_edges` lets a source route to one or more destinations or `END` through a path function (`/git/github.com/langchain-ai/langgraph/libs/langgraph/langgraph/graph/state.py:891-930`). `Send` sends custom state to a target node, supporting map-reduce-style fanout (`/git/github.com/langchain-ai/langgraph/libs/langgraph/langgraph/types.py:654-698`). `Command` can update state, resume interrupts, and navigate to nodes or `Send`s (`/git/github.com/langchain-ai/langgraph/libs/langgraph/langgraph/types.py:748-769`). `interrupt` pauses graph execution and resumes through `Command`, with a checkpointer requirement (`/git/github.com/langchain-ai/langgraph/libs/langgraph/langgraph/types.py:801-820`).

LangChain docs classify multi-agent patterns as subagents, handoffs, skills, router, and custom workflow, and define custom workflow as using LangGraph for bespoke deterministic plus agentic flow (LangChain, "Multi-agent", accessed 2026-05-04, https://docs.langchain.com/oss/python/langchain/multi-agent/index). The LangGraph overview describes it as low-level orchestration for long-running, stateful agents with durable execution, streaming, and human-in-the-loop support (LangChain, "LangGraph overview", accessed 2026-05-04, https://docs.langchain.com/oss/python/langgraph).

Architectural take: LangGraph is less a council pattern than the missing explicit state-machine layer under many council workflows.

## Claude Code Subagents and Plugins

The public Claude Code docs define subagents as specialized assistants with their own context window, system prompt, tool access, and independent permissions, delegated to by Claude when a task matches their description (Anthropic, "Create custom subagents", accessed 2026-05-04, https://code.claude.com/docs/en/subagents). The same docs name built-in Explore, Plan, and general-purpose subagents and describe Explore/Plan as read-only codebase research agents (Anthropic, "Create custom subagents", accessed 2026-05-04, https://code.claude.com/docs/en/subagents). They also document parallel research, chaining, main-conversation versus subagent choice, and the fact that subagents cannot spawn other subagents (Anthropic, "Create custom subagents", accessed 2026-05-04, https://code.claude.com/docs/en/subagents).

The cloned Claude Code plugin examples show practical ensemble patterns. The plugin README says plugins extend Claude Code with custom commands, specialized agents, hooks, skills, and MCP servers (`/git/github.com/anthropics/claude-code/plugins/README.md:3-9`, `/git/github.com/anthropics/claude-code/plugins/README.md:47-60`). The code-review plugin table describes five parallel Sonnet agents for compliance, bug detection, historical context, PR history, and comments (`/git/github.com/anthropics/claude-code/plugins/README.md:17-18`). The feature-dev plugin launches parallel code explorers, parallel architects, and parallel reviewers (`/git/github.com/anthropics/claude-code/plugins/feature-dev/commands/feature-dev.md:36-53`, `/git/github.com/anthropics/claude-code/plugins/feature-dev/commands/feature-dev.md:73-82`, `/git/github.com/anthropics/claude-code/plugins/feature-dev/commands/feature-dev.md:101-109`). The code-review command launches multiple agents, validates flagged issues with follow-up subagents, filters unvalidated issues, and outputs or posts only high-signal findings (`/git/github.com/anthropics/claude-code/plugins/code-review/commands/code-review.md:24-77`).

Architectural take: Claude Code offers the most relevant practical pattern for Criopolis tool agents: independent subagents for bounded sidecar work, then main-agent synthesis.

## Anthropic Multi-Agent Research System

Anthropic describes its Research architecture as an orchestrator-worker pattern: a lead agent coordinates specialized subagents operating in parallel (Anthropic, "How we built our multi-agent research system", 2025-06-13, https://www.anthropic.com/engineering/multi-agent-research-system). The blog says the lead agent analyzes a query, develops a strategy, spawns subagents, each subagent searches/evaluates independently, and the lead agent synthesizes results and may create more subagents before final citation processing (Anthropic, "How we built our multi-agent research system", 2025-06-13, https://www.anthropic.com/engineering/multi-agent-research-system). It reports internal findings that the architecture is especially useful for breadth-first queries, independent directions, large context, and many tools, while tasks with many dependencies or shared-context needs are a poor fit today (Anthropic, "How we built our multi-agent research system", 2025-06-13, https://www.anthropic.com/engineering/multi-agent-research-system).

Anthropic's separate "Building effective agents" post distinguishes workflows, where LLMs/tools follow predefined code paths, from agents, where LLMs dynamically direct process and tool use (Anthropic, "Building effective agents", 2024-12-19, https://www.anthropic.com/engineering/building-effective-agents). The same post says successful implementations often use simple composable patterns rather than complex frameworks (Anthropic, "Building effective agents", 2024-12-19, https://www.anthropic.com/engineering/building-effective-agents).

Architectural take: this is the strongest production argument for research fanout plus lead-agent synthesis, and it supports Criopolis's choice to give researcher work separate contexts before mayor/council synthesis.

## OpenAI Swarm

Swarm is a lightweight handoff framework. Its README says it is experimental/educational and replaced by the OpenAI Agents SDK for production (`/git/github.com/openai/swarm/README.md:3-8`). It describes lightweight coordination via agents and handoffs, with stateless client-side Chat Completions execution (`/git/github.com/openai/swarm/README.md:72-88`). In source, an `Agent` has name, model, instructions, functions, tool choice, and parallel tool call setting; a `Result` can carry value, next agent, and context variables (`/git/github.com/openai/swarm/swarm/types.py:14-41`). `get_chat_completion` builds a system message from the active agent, converts functions into tools, hides context variables from the model schema, and calls chat completions (`/git/github.com/openai/swarm/swarm/core.py:32-69`). `handle_function_result` turns a returned `Agent` into a `Result(agent=agent)` (`/git/github.com/openai/swarm/swarm/core.py:71-87`). `handle_tool_calls` executes mapped functions, records tool messages, merges context variables, and stores next agent if present (`/git/github.com/openai/swarm/swarm/core.py:89-137`). `run` loops completions and tool calls and switches active agent when a tool result returns one (`/git/github.com/openai/swarm/swarm/core.py:231-292`).

Architectural take: Swarm is the cleanest minimal code reference for handoff-as-tool-result.

## CAMEL

CAMEL has both role-playing societies and workforce orchestration. `RolePlaying` is a two-agent setup between assistant and user roles, with optional critic, task specifier, task planner, task type, model, and other agent arguments (`/git/github.com/camel-ai/camel/camel/societies/role_playing.py:36-90`). Its `step` advances by passing an assistant message to the user agent, reducing the selected user response, then passing that user message to the assistant agent, reducing the selected assistant response, and returning both responses (`/git/github.com/camel-ai/camel/camel/societies/role_playing.py:631-710`). Workforce is a multi-worker system that can assign tasks, create workers, decompose tasks, and recover from failures; its internal specialized agents are Coordinator, Task Planner, and Dynamic Workers (`/git/github.com/camel-ai/camel/camel/societies/workforce/workforce.py:175-186`). Workforce can share memory among SingleAgentWorkers, coordinator, and task planner (`/git/github.com/camel-ai/camel/camel/societies/workforce/workforce.py:224-231`). It can also add role-playing workers (`/git/github.com/camel-ai/camel/camel/societies/workforce/workforce.py:3071-3095`), and role-playing workers create role-playing sessions, alternate steps until turn limit/termination/done, then summarize chat history into a task result (`/git/github.com/camel-ai/camel/camel/societies/workforce/role_playing_worker.py:103-235`).

The CAMEL paper frames communicative agents as a scalable approach to studying cooperative behaviors in LLM societies (Li et al., "CAMEL: Communicative Agents for 'Mind' Exploration of Large Language Model Society", 2023-03-31, https://arxiv.org/abs/2303.17760).

Architectural take: CAMEL is the closest surveyed codebase to explicit role-playing and "society" research, but its primary primitive is conversation/workforce, not durable civic governance.

## Criopolis / Gas City

Criopolis defines itself through prompts and bead/mail operations. `pack.toml` names mayor, five council seats, researcher, explorer, code-writer, majordomo, and named sessions (`pack.toml:5-118`). Mayor orchestrates high council, tool agents, and majordomo; the five seats are Satya, Viveka, Dharma, Prayoga, and Rasa (`agents/mayor/prompt.template.md:73-104`). Mayor's loop is frame, sling, watch closes, push back, synthesize (`agents/mayor/prompt.template.md:109-130`). Majordomo is always-on and holds cascade receiver, state tracker, routine dispatcher, and first-pass synthesizer responsibilities (`agents/majordomo/prompt.template.md:3-12`, `agents/majordomo/prompt.template.md:34-108`). Researcher interacts with beads through `bd`, clones source with `ghq get`, writes answers under `research/answers/`, updates bead notes, closes, and mails majordomo (`agents/researcher/prompt.template.md:27-44`, `agents/researcher/prompt.template.md:74-117`).

Architectural take: Criopolis is a durable civic workflow with a deliberative council. The closest pieces in prior systems are Anthropic's lead/subagent research pattern, AutoGen's group-chat manager, LangGraph's state graph, Claude Code's parallel subagent workflows, and CAMEL's role-playing societies. The exact blend of durable beads, mayor/majordomo split, and five philosophical council seats was not found in the surveyed source.
