# cr-607sxs (bead-close to git-commit linkage)

## Headline finding

Criopolis can make bead close trigger a source `git commit`, but the source-backed path is a custom script at a bead or Gas City trigger point, not a built-in Gas City "close bead -> commit repo" feature. Gas City has a `bead.closed` event type (`/git/github.com/gastownhall/gascity/internal/events/events.go:18-24`) and event orders can fire on `on = "bead.closed"` (`/git/github.com/gastownhall/gascity/docs/tutorials/07-orders.md:198-213`), but the commit action itself is only a shell `exec`/formula/script pattern (`/git/github.com/gastownhall/gascity/internal/orders/order.go:19-24`, `/git/github.com/gastownhall/gascity/docs/tutorials/07-orders.md:230-256`).

The best exact-ID integration point is Beads' `.beads/hooks/on_close`: Beads defines `on_close` as the hook file for close events (`/git/github.com/gastownhall/beads/internal/hooks/hooks.go:13-25`, `/git/github.com/gastownhall/beads/internal/hooks/hooks.go:125-135`), wraps write stores so close mutations fire the close hook (`/git/github.com/gastownhall/beads/internal/storage/hook_decorator.go:108-115`), and executes hook scripts as `hook_script <issue_id> <event_type>` with the full issue JSON on stdin (`/git/github.com/gastownhall/beads/internal/hooks/hooks_unix.go:46-55`). That avoids guessing which bead closed.

## Bead structure recap

A bead has stable `ID` plus separate `Title`, `Description`, `Notes`, and custom `Metadata`; those are distinct fields in the issue type (`/git/github.com/gastownhall/beads/internal/types/types.go:14-27`, `/git/github.com/gastownhall/beads/internal/types/types.go:56-59`). `bd create` requires a title from either the positional argument or `--title` (`/git/github.com/gastownhall/beads/cmd/bd/create.go:27-87`), while description is loaded separately and may be empty unless config requires it (`/git/github.com/gastownhall/beads/cmd/bd/create.go:103-127`). `bd update` can later change title and description independently (`/git/github.com/gastownhall/beads/cmd/bd/update.go:84-101`).

Closing a bead is a status transition, not a source commit. `bd close` resolves IDs and calls `store.CloseIssue(...)` (`/git/github.com/gastownhall/beads/cmd/bd/close.go:111-190`), and the Dolt storage implementation sets `status`, `closed_at`, `updated_at`, `close_reason`, and `closed_by_session` (`/git/github.com/gastownhall/beads/internal/storage/dolt/transaction.go:597-614`). Beads' own Dolt commit command commits the Beads database working set, with a default message like `bd: dolt commit (auto-commit) by <actor>`; that is Beads data history, not a source-code commit (`/git/github.com/gastownhall/beads/cmd/bd/dolt.go:321-345`). Beads also documents that it works without git and that `--stealth` disables git hook installation and git operations (`/git/github.com/gastownhall/beads/README.md:156-176`).

## Patterns found

Gas City's minimal work formula tells the agent to make normal git commits while doing the work, then close the bead afterward (`/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-do-work.toml:40-52`). The fuller core worker lifecycle uses commit messages with the bead ID in parentheses, `git commit -m "<type>: <description> ({{issue}})"`, and repeats that pattern for uncommitted leftovers during final review (`/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-base.toml:163-179`, `/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-base.toml:238-251`).

The core direct-commit variant is commit-before-close, not close-before-commit: it creates a bead-scoped worktree, stores `work_dir`, commits any remaining changes, pushes to the base branch, removes the worktree, updates notes, and then closes the bead (`/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-commit.toml:1-17`, `/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-commit.toml:62-70`, `/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-commit.toml:95-138`, `/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-commit.toml:150-151`).

The Kit PR workflow follows the same "git first, bead handoff later" pattern: the pack says the done sequence is tests -> push -> create PR -> hand bead to steward (`/git/github.com/bmt/gascity-explore/packs/kit/pack.toml:1-11`), its formula commits leftovers, pushes, creates a PR whose body starts with `Resolves {{issue}}`, records PR/branch metadata, assigns the bead to the steward, and exits (`/git/github.com/bmt/gascity-explore/packs/kit/formulas/mol-polecat-pr.toml:1-17`, `/git/github.com/bmt/gascity-explore/packs/kit/formulas/mol-polecat-pr.toml:50-82`, `/git/github.com/bmt/gascity-explore/packs/kit/formulas/mol-polecat-pr.toml:93-122`). Its template fragment also shows ordinary `git add && git commit`, push, PR create, then bead metadata updates (`/git/github.com/bmt/gascity-explore/packs/kit/template-fragments/git-workflow-pr.template.md:1-44`).

The Discord pack records bead-scoped worktree and branch metadata (`work_dir`, `branch`) and reports bead/branch/worktree on completion, but the fallback completion path removes the worktree and closes the bead rather than explicitly creating a source commit there (`/git/github.com/gastownhall/gascity-packs/discord/formulas/mol-discord-fix-issue.formula.toml:1-16`, `/git/github.com/gastownhall/gascity-packs/discord/formulas/mol-discord-fix-issue.formula.toml:121-164`, `/git/github.com/gastownhall/gascity-packs/discord/formulas/mol-discord-fix-issue.formula.toml:247-264`, `/git/github.com/gastownhall/gascity-packs/discord/formulas/mol-discord-fix-issue.formula.toml:283-323`). The UBS pack hooks `PreToolUse` to scan staged files when the agent runs `git commit`; it does not initiate the commit itself (`/git/github.com/gastownhall/gascity-packs/flywheel/ubs/README.md:1-8`).

Gas City's Git helper package is for repo/worktree status and safety operations: repo detection, branch lookup, worktree list/remove, uncommitted/unpushed checks, stash, fetch, pull-rebase, status, and ahead/behind (`/git/github.com/gastownhall/gascity/internal/git/git.go:1-240`). I found no native source-code commit helper or bead-close commit path in the inspected Gas City sources; the inspected code points to formula/script-level commits instead.

## Bead-id-as-commit-message viability

Using the bead ID in the commit message is viable and already matches local formulas: core uses `({{issue}})` in commit subjects (`/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-base.toml:173-179`, `/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-base.toml:244-247`), Kit PR bodies use `Resolves {{issue}}` (`/git/github.com/bmt/gascity-explore/packs/kit/formulas/mol-polecat-pr.toml:64-82`), and the direct-commit formula stores bead metadata before closing (`/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-commit.toml:62-70`, `/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-commit.toml:134-138`).

A bare `<bead-id>` is stable but opaque in `git log --oneline`. The stronger local convention is either `<type>: <description> (<bead-id>)`, as the core formulas already prescribe (`/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-base.toml:173-179`), or `<bead-id>: <title>` when the requirement is bead-first lookup. Because `Title` is the short required field and `Description`/`Notes` are longer separate fields, `<bead-id>: <title>` is a good automatic subject source (`/git/github.com/gastownhall/beads/internal/types/types.go:21-27`, `/git/github.com/gastownhall/beads/cmd/bd/create.go:64-87`).

## Recommendation

Use one of two mechanisms, depending on how exact the linkage must be:

1. Prefer a Beads `on_close` hook for exact automatic close -> commit. The hook receives the exact bead ID as argv and full issue JSON on stdin (`/git/github.com/gastownhall/beads/internal/hooks/hooks_unix.go:46-55`), close hooks are fired by the storage decorator after close mutations (`/git/github.com/gastownhall/beads/internal/storage/hook_decorator.go:108-115`, `/git/github.com/gastownhall/beads/internal/storage/hook_decorator.go:193-209`), and all mutations inside Beads transactions fire hooks only after commit succeeds (`/git/github.com/gastownhall/beads/internal/storage/hook_decorator.go:171-188`). The script should parse the issue JSON, derive `repo` from `metadata.work_dir` or a configured Criopolis rig path, check `git status --porcelain`, refuse unsafe states, then run `git add -A && git commit -m "$id: $title"` or `git commit -m "$type: $title ($id)"`, and finally record the commit SHA back to the bead notes or metadata.

2. Use a Gas City event order only when a controller-managed, coarse-grained runner is acceptable. Gas City emits/recognizes `bead.closed` (`/git/github.com/gastownhall/gascity/internal/events/events.go:18-24`), tests show bead create/close operations should be visible via `gc events` (`/git/github.com/gastownhall/gascity/test/integration/gastown_events_test.go:10-29`), and orders can run direct shell `exec` commands on event triggers (`/git/github.com/gastownhall/gascity/docs/tutorials/07-orders.md:198-213`, `/git/github.com/gastownhall/gascity/docs/tutorials/07-orders.md:230-256`). The order would be `[order] trigger = "event"; on = "bead.closed"; exec = "$ORDER_DIR/scripts/bead-autocommit.sh"`.

If choosing the Gas City order route, the script must compensate for the dispatcher shape. Events have `Subject` and `Payload` fields (`/git/github.com/gastownhall/gascity/internal/events/events.go:99-108`), and cache notifications record the bead ID as subject with bead JSON payload (`/git/github.com/gastownhall/gascity/internal/beads/caching_store_events.go:335-345`, `/git/github.com/gastownhall/gascity/cmd/gc/api_state.go:124-131`). But event triggers only check `Type` and `AfterSeq` (`/git/github.com/gastownhall/gascity/internal/orders/triggers.go:175-195`), the event filter only supports `Type`, `Actor`, `Since`, and `AfterSeq` (`/git/github.com/gastownhall/gascity/internal/events/reader.go:13-19`, `/git/github.com/gastownhall/gascity/internal/events/reader.go:49-75`), and the order dispatcher uses the trigger result only to decide due before launching the order (`/git/github.com/gastownhall/gascity/cmd/gc/order_dispatch.go:202-254`). Exec orders receive store/order environment like `GC_STORE_ROOT`, `GC_RIG_ROOT`, `ORDER_DIR`, and pack paths, not the matched event object (`/git/github.com/gastownhall/gascity/cmd/gc/order_store.go:91-128`, `/git/github.com/gastownhall/gascity/cmd/gc/order_dispatch.go:339-370`).

The safest operational pattern is still commit-before-close where possible: the core direct-commit formula does final status, commits leftovers, pushes, cleans the worktree, updates notes, then closes (`/git/github.com/gastownhall/gascity/internal/bootstrap/packs/core/formulas/mol-polecat-commit.toml:95-138`). A small Criopolis `finish-bead <id>` wrapper that performs commit then `bd close` would follow that source-backed pattern and avoid asynchronous hook failure after the bead is already closed.

## What gc cannot do here

`gc hook --inject` is not the answer: the command documents `--inject` as silent legacy Stop-hook compatibility that skips work query and always exits 0 (`/git/github.com/gastownhall/gascity/cmd/gc/cmd_hook.go:18-56`), tests assert fresh hook configs should not install no-op `gc hook --inject` (`/git/github.com/gastownhall/gascity/internal/hooks/hooks_test.go:124-132`, `/git/github.com/gastownhall/gascity/internal/hooks/hooks_test.go:770-783`), and the changelog says fresh managed configs no longer install it (`/git/github.com/gastownhall/gascity/CHANGELOG.md:23-26`).

Gas City can trigger work on `bead.closed`, but it cannot natively infer "the closed bead corresponds to this source repo and these dirty files" from the order trigger alone: order events are matched by event type/cursor (`/git/github.com/gastownhall/gascity/internal/orders/triggers.go:175-195`), filterable event fields omit subject/payload/repo path (`/git/github.com/gastownhall/gascity/internal/events/reader.go:13-19`), and exec orders run a configured shell command with environment rather than a passed matched event (`/git/github.com/gastownhall/gascity/cmd/gc/order_dispatch.go:339-370`, `/git/github.com/gastownhall/gascity/cmd/gc/order_store.go:91-128`).

Gas City also cannot make a source commit without a custom command/script/formula. Its order model supports either `formula` or `exec`, with `exec` explicitly being a shell command (`/git/github.com/gastownhall/gascity/internal/orders/order.go:19-24`, `/git/github.com/gastownhall/gascity/internal/orders/order.go:140-152`), and its Git helper package covers status/worktree/fetch/stash/rebase operations rather than a bead-close commit primitive (`/git/github.com/gastownhall/gascity/internal/git/git.go:1-240`).
