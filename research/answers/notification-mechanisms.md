# Notification mechanisms deep dive

## 1. Claude Code hook guarantees

Claude Code documents these as lifecycle events: `SessionStart` when a session begins/resumes, `UserPromptSubmit` before Claude processes a submitted prompt, `Stop` when Claude finishes responding, and `PreCompact` before compaction; `PreCompact` distinguishes `manual` and `auto` matchers, and `SessionStart` distinguishes `startup`, `resume`, `clear`, and `compact` sources (Claude Hooks reference, "Hook lifecycle" and "SessionStart"/"PreCompact": https://code.claude.com/docs/en/hooks#hook-lifecycle, https://code.claude.com/docs/en/hooks#sessionstart, https://code.claude.com/docs/en/hooks#precompact). Matching Claude hook handlers do not have serial ordering semantics: the docs say matching hooks run in parallel and identical handlers are deduplicated (Claude Hooks reference, "Prompt and agent hook fields": https://code.claude.com/docs/en/hooks#prompt-and-agent-hook-fields). Command hooks receive event JSON on stdin and communicate by exit code/stdout/stderr; common fields include `session_id`, `transcript_path`, `cwd`, `permission_mode`, and `hook_event_name` (Claude Hooks reference, "Hook input and output": https://code.claude.com/docs/en/hooks#hook-input-and-output). Exit 0 parses stdout JSON; plain stdout reaches Claude as context for `SessionStart` and `UserPromptSubmit`, while `Stop` and `PreCompact` can block through `decision: "block"`/`reason` or exit code 2 (Claude Hooks reference, "Exit code output", "Decision control", "UserPromptSubmit", "Stop", "PreCompact": https://code.claude.com/docs/en/hooks#exit-code-output). Timeouts are configurable in seconds; defaults are 600 for command hooks, 30 for prompt hooks, and 60 for agent hooks (Claude Hooks reference, "Hook handler fields": https://code.claude.com/docs/en/hooks#hook-handler-fields). I found no retry guarantee for these four events in the hook reference; `retry` appears as a different `PermissionDenied` decision field, not a lifecycle hook retry (Claude Hooks reference, "Decision control": https://code.claude.com/docs/en/hooks#decision-control). Idle/background is separate: `TeammateIdle` is its own event, and `async: true` background hooks cannot control the event they came from; async output is delivered later if it includes context (Claude Hooks reference, "Hook lifecycle" and "Run hooks in the background": https://code.claude.com/docs/en/hooks#run-hooks-in-the-background).

## 2. Codex hook guarantees

The current Codex source only enumerates six hook events: `PreToolUse`, `PermissionRequest`, `PostToolUse`, `SessionStart`, `UserPromptSubmit`, and `Stop`; there is no `PreCompact` hook in the protocol enum or hook config struct (`/git/github.com/openai/codex/codex-rs/protocol/src/protocol.rs:1513-1522`, `/git/github.com/openai/codex/codex-rs/config/src/hook_config.rs:31-44`). Codex local docs expose `hooks/list` and show hook metadata including event name, command, `timeoutSec`, source path, source, and display order (`/git/github.com/openai/codex/codex-rs/app-server/README.md:1460-1492`); the Codex README says new automation should use lifecycle hooks instead of legacy `notify` (`/git/github.com/openai/codex/codex-rs/README.md:47-49`).

Codex selects handlers in config/display order but executes all selected handlers concurrently via `join_all`, so there is no serial invocation guarantee (`/git/github.com/openai/codex/codex-rs/hooks/src/engine/dispatcher.rs:42-61`, `/git/github.com/openai/codex/codex-rs/hooks/src/engine/dispatcher.rs:83-103`). `SessionStart` matchers receive `startup`/`resume`/`clear`; `UserPromptSubmit` and `Stop` ignore matchers (`/git/github.com/openai/codex/codex-rs/hooks/src/events/session_start.rs:20-34`, `/git/github.com/openai/codex/codex-rs/hooks/src/events/common.rs:98-108`). Command hooks are spawned once with piped stdin/stdout/stderr, receive serialized JSON on stdin, and are killed on timeout; default timeout is 600 seconds unless `timeout_sec` is configured (`/git/github.com/openai/codex/codex-rs/hooks/src/engine/command_runner.rs:33-39`, `/git/github.com/openai/codex/codex-rs/hooks/src/engine/command_runner.rs:56-99`, `/git/github.com/openai/codex/codex-rs/hooks/src/engine/discovery.rs:379-403`). I found no retry path in hook execution; `run_command` writes stdin once and waits once (`/git/github.com/openai/codex/codex-rs/hooks/src/engine/command_runner.rs:56-99`). Input schemas are explicit for `SessionStart`, `UserPromptSubmit`, and `Stop` (`/git/github.com/openai/codex/codex-rs/hooks/src/schema.rs:335-405`). `UserPromptSubmit` accepts JSON or plain-text stdout as additional context and exit 2 as a block; `Stop` accepts JSON/exit-2 blocking with continuation text (`/git/github.com/openai/codex/codex-rs/hooks/src/events/user_prompt_submit.rs:147-228`, `/git/github.com/openai/codex/codex-rs/hooks/src/events/stop.rs:145-222`). The TUI descriptions bind these events to new session, prompt submit, and turn stop, with no idle/background hook event (`/git/github.com/openai/codex/codex-rs/tui/src/bottom_pane/hooks_browser_view.rs:545-552`).

## 4. Authoritative "agent done" event today

Use bead close as the source of truth for "agent A finished work-bead X". Bead close is the actual state transition: the Store interface defines `Close` as setting status to closed and idempotent on already-closed beads, and the bd-backed implementation calls `bd close --force --json` (`/git/github.com/gastownhall/gascity/internal/beads/beads.go:147-149`, `/git/github.com/gastownhall/gascity/internal/beads/bdstore.go:751-767`). Gas City then forwards bd `on_close` to `bead.closed` with the closed bead ID as subject (`/git/github.com/gastownhall/gascity/cmd/gc/hooks.go:11-16`, `/git/github.com/gastownhall/gascity/cmd/gc/hooks.go:34-55`). The event bus defines `bead.closed`, can be watched with `gc events --watch --type ...`, and event-triggered orders read only events after a cursor (`/git/github.com/gastownhall/gascity/internal/events/events.go:18-27`, `/git/github.com/gastownhall/gascity/cmd/gc/cmd_events.go:123-180`, `/git/github.com/gastownhall/gascity/internal/orders/triggers.go:175-195`). Caveat: event recording is best-effort and errors are not returned, so the durable truth remains the bead status; the event is the subscription surface (`/git/github.com/gastownhall/gascity/internal/events/events.go:1-9`, `/git/github.com/gastownhall/gascity/internal/events/recorder.go:55-94`). Mail is a durable notification, but it is a separate message bead, not proof that work bead X was closed (`/git/github.com/gastownhall/gascity/internal/mail/beadmail/beadmail.go:37-72`).

## 5. On-demand materialization on nudge

For `gc session nudge <name>`, the CLI resolves the target before delivery, and resolution explicitly calls `resolveSessionIDMaterializingNamed` (`/git/github.com/gastownhall/gascity/cmd/gc/cmd_session.go:1619-1630`, `/git/github.com/gastownhall/gascity/cmd/gc/cmd_nudge.go:607-630`). That resolver reopens a closed configured named session or calls `ensureSessionIDForTemplateWithOptions` when materialization is allowed (`/git/github.com/gastownhall/gascity/cmd/gc/session_resolve.go:67-85`, `/git/github.com/gastownhall/gascity/cmd/gc/session_resolve.go:96-98`). Managed cities create a deferred session bead and poke the controller; non-managed materialization starts the runtime immediately (`/git/github.com/gastownhall/gascity/cmd/gc/session_template_start.go:184-201`, `/git/github.com/gastownhall/gascity/cmd/gc/session_template_start.go:215-228`). The worker handle also creates a deferred session ID if one is missing before nudging (`/git/github.com/gastownhall/gascity/internal/worker/handle_lifecycle.go:301-390`). Default `wait-idle` delivery tries live delivery; if not delivered, the CLI queues the nudge (`/git/github.com/gastownhall/gascity/cmd/gc/cmd_nudge.go:491-518`). Runtime-only handles are weaker: they require an already-running runtime unless live-only returns undelivered (`/git/github.com/gastownhall/gascity/internal/worker/runtime_handle.go:228-269`). Provider delivery is best-effort: tmux ignores missing sessions, ACP returns nil for missing/dead sessions and errors on busy timeout, exec delegates to a script, and subprocess nudge is unsupported/no-op (`/git/github.com/gastownhall/gascity/internal/runtime/tmux/adapter.go:327-387`, `/git/github.com/gastownhall/gascity/internal/runtime/acp/acp.go:456-486`, `/git/github.com/gastownhall/gascity/internal/runtime/exec/exec.go:411-420`, `/git/github.com/gastownhall/gascity/internal/runtime/subprocess/subprocess.go:263-267`).

## 6. Cascade primitive recommendation

The clean primitive is: A writes final notes and closes the work bead, then sends B mail with notify. Shape:

1. `bd update <work-bead> --notes "..."; bd close <work-bead>`
2. `gc mail send --notify <agent-b> -s "done: <work-bead>" -m "A closed <work-bead>; summary..."`

This uses bead close as completion truth and mail as durable attention. `gc mail send` creates a message, emits `mail.sent`, and only then optionally nudges the recipient; nudge failure is non-fatal stderr, so the mail remains the record (`/git/github.com/gastownhall/gascity/cmd/gc/cmd_mail.go:1001-1041`, `/git/github.com/gastownhall/gascity/cmd/gc/cmd_mail.go:1069-1090`). B sees a live or queued "You have mail from A" nudge when possible; otherwise B sees unread mail when the `UserPromptSubmit` hooks run `gc nudge drain --inject` and `gc mail check --inject` (`/git/github.com/gastownhall/gascity/cmd/gc/cmd_nudge.go:572-604`, `/home/li/Criopolis/.gc/settings.json:37-50`, `/home/li/Criopolis/.codex/hooks.json:25-39`). Mail check injection always exits 0 and formats unread mail as hook context, so hook failures do not block the provider turn (`/git/github.com/gastownhall/gascity/cmd/gc/cmd_mail.go:159-218`, `/git/github.com/gastownhall/gascity/cmd/gc/cmd_mail.go:229-247`). No new Claude/Codex config is needed for this city; removing stale `Stop` hooks and adding missing Claude `--auto` are cleanup, not prerequisites.

## Recommended cascade primitive

Build the minimum primitive around `bd close` plus `gc mail send --notify`. Do not use provider `Stop`, handoff, or `session.stopped` as the completion source: provider hooks are lifecycle/turn hooks, `gc hook --inject` is a no-op compatibility shim, and session lifecycle does not encode which work bead was completed (`/git/github.com/gastownhall/gascity/cmd/gc/cmd_hook.go:24-28`, `/git/github.com/gastownhall/gascity/cmd/gc/cmd_hook.go:53-59`). The implementation should treat event-log `bead.closed` as a convenient subscription and always be able to verify the bead's closed status if exact durability matters (`/git/github.com/gastownhall/gascity/internal/events/recorder.go:55-94`, `/git/github.com/gastownhall/gascity/internal/beads/bdstore.go:751-767`).
